# 苍穹研发问题原因分析

## 适用场景

- 生产环境故障排查
- 系统性能问题分析
- 代码Bug根因定位
- 部署/发布问题诊断
- 异常告警分析

## 分析流程

### 第一步：问题定义

明确以下信息：
- **现象描述**：用户看到什么错误？系统表现如何异常？
- **影响范围**：哪些功能/模块/用户受影响？
- **时间信息**：何时开始？是否间歇性？
- **触发条件**：什么操作会导致问题？

### 第二步：数据收集

根据问题类型收集对应数据源：

**日志数据**：
- 应用日志（error/warn级别）
- 系统日志（/var/log/）
- 访问日志（nginx/apache）
- 数据库慢查询日志

**监控数据**：
- 性能指标（CPU/内存/磁盘/网络）
- 链路追踪（Jaeger/SkyWalking）
- 告警历史
- 业务指标趋势

**代码/配置**：
- 近期代码变更（git log）
- 配置文件变更
- 依赖版本变化
- 环境差异对比

### 第三步：根因分析

使用 5 Whys 方法逐层深入：

```
问题现象
    ↓ 为什么？
直接原因
    ↓ 为什么？
中间原因
    ↓ 为什么？
根本原因
```

常见根因分类：
- **代码缺陷**：逻辑错误、边界条件、并发问题
- **配置问题**：参数错误、环境差异、配置漂移
- **资源问题**：内存泄漏、连接池耗尽、磁盘满
- **依赖问题**：第三方服务故障、版本不兼容
- **数据问题**：脏数据、数据迁移、索引缺失

### 第四步：验证假设

对每个可能原因进行验证：
- [ ] 复现问题
- [ ] 查看相关日志证据
- [ ] 检查监控指标关联性
- [ ] 对比正常/异常环境

## 常用命令参考

**日志搜索**：
```bash
# 按关键词搜索
grep -i "error\|exception" app.log

# 按时间范围搜索
awk '/2024-01-15 14:30/,/2024-01-15 15:00/' app.log

# 实时跟踪
tail -f app.log | grep ERROR
```

**性能分析**：
```bash
# 查看资源使用
top / htop

# 线程Dump
jstack <pid> > thread_dump.txt

# 内存分析
jmap -dump:format=b,file=heap.hprof <pid>
```

**Git分析**：
```bash
# 近期变更
git log --since="1 day ago" --oneline

# 文件变更历史
git log -p --follow <file>

# 某行代码最后修改
git blame <file>
```

## 分析技巧

1. **时间线对齐**：将日志、监控、告警按时间轴对齐，寻找关联性
2. **对比分析**：对比正常环境与异常环境的差异
3. **分层排查**：从应用层 → 框架层 → 系统层 → 网络层逐层排查
4. **假设驱动**：先提出假设，再寻找证据验证或证伪
5. **关注变化**：问题发生前有什么变更？变更往往是问题的源头

## 案例二十二详情：租户中心数据未发布到ZK

> 来源：https://vip.kingdee.com/link/s/ZIBj5

### 问题描述

报错信息：**租户中心数据未发布到ZK，请在配置管理中心发布相关配置项**

根本原因是MC（租户管理中心）配置的租户数据未发布到ZK（ZooKeeper），导致苍穹服务从ZK获取不到租户相关数据，进而引发系统异常。

### 原因分析

该问题的可能原因有以下三种：

1. **修改租户编码后未发布集群**
   - 在MC上修改了租户编码，但未执行集群发布操作，导致ZK中的配置数据未及时更新。

2. **本地开发环境租户编码配置错误**
   - 本地开发时，启动参数或配置文件中 `domain.tenantCode` 的值与MC上配置的租户编码不一致，导致无法正确匹配租户数据。

3. **数据中心未启用**
   - MC中对应的数据中心处于未启用状态，租户数据未生效，自然也就无法同步到ZK。

### 解决方案

针对不同原因，按以下步骤排查和修复：

**情况一：MC配置未发布到ZK**

1. 登录MC管理控制台。
2. 执行**发布集群**操作，将最新的租户配置数据推送至ZK。
3. 发布完成后，登录 **Monitor【注册中心】→【系统属性】**。
4. 搜索 `mc.tenant.租户编码.data`，确认数据已正确发布到ZK。
5. 若数据已存在但问题仍未解决，需进一步分析其他原因。

**情况二：本地开发租户编码不一致**

1. 检查本地开发环境的启动参数或配置文件中 `domain.tenantCode` 的值。
2. 比对该值与MC上配置的租户编码是否一致。
3. 若不一致，修改 `domain.tenantCode` 对应的 value 值，使其与MC保持一致。

**情况三：数据中心未启用**

1. 登录MC管理控制台。
2. 进入**租户列表**。
3. 找到对应的数据中心，将其状态设置为**启用**。
4. 保存后执行**发布集群**操作，使配置生效。

## 案例二十三详情：苍穹启动报错NoClassDefFoundError

> 来源：https://vip.kingdee.com/link/s/ZIB1I

### 问题描述

苍穹启动时报异常：

```
java.lang.NoClassDefFoundError: Could not initialize class org.apache.poi.ooxml.POIXMLTypeLoader
```

### 原因分析

`NoClassDefFoundError` 属于常见的JAR包缺失或JAR包版本不匹配类错误。在分析原因前，可先根据异常类型缩小排查范围：

- **ClassNotFoundException**：类找不到，应确认对应的JAR包是否已引入。
- **FileNotFoundException**：文件找不到，应确认文件路径是否正确、文件是否存在。
- **NullException**：空指针异常，检查参数在什么情况下可能为空值。
- **NoClassDefFoundError / MethodNotFoundException**：JAR包未引入或JAR包版本不对应。

本次报错的类 `org.apache.poi.ooxml.POIXMLTypeLoader` 属于第三方工具类（Apache POI），因此需从JAR包引入和版本方向排查。

### 解决方案

苍穹的JAR包分为4类：

- **bos**：苍穹核心包，包名以 `kd.bos` 开头。
- **biz**：星瀚应用包，包名以 `kd` 开头，第二三级包名是云编码/应用编码。
- **trd**：第三方插件包，例如 apache、log4j 等，包名非 `kd` 开头。
- 其他非标准包。

**排查步骤**：

1. 根据报错类名判断所属JAR包。若是第三方插件，可拿类名百度搜索找到对应JAR包；若是苍穹的包，可通过 Eclipse/IDEA 等开发工具搜索类，查看所属JAR。
2. 本次错误提示的类属于 `trd/poi-ooxml-4.1.2.jar`。
3. 检查苍穹服务下 `trd` 目录中该JAR是否存在、版本是否正确。
4. 若JAR缺失或版本错误，替换为正确版本后重启服务。

**Tips**：此类错误更多可能是苍穹服务做过调整（例如补丁升级未完成）。遇到该错误时，应回想出问题前做过哪些操作，是否影响到苍穹的JAR包。

## 案例二十四详情：MC升级校验异常问题汇总

> 来源：https://vip.kingdee.com/link/s/ZIBwE

### 问题一：【MC-008-003】当前集群为金蝶AI星瀚，请确认版本是否正常

**现象**：升级补丁时报错，提示当前集群为金蝶AI星瀚，请确认版本是否正常。

**解决方法**（一般直接执行第3步即可）：

1. 在MC所在数据库实例执行SQL：`select * from t_mc_version_mutex_rule;`，确认 `fmodetype` 的值为 `",galaxy,constellation,"`。
2. 如果成立，执行删除语句：`delete from t_mc_datacenterversions where fregion = 'constellation';`
3. 如果仍然失败，到MC数据中心找到**元数据库**的实例，执行SQL：`delete from t_bas_industry_version where fproductnumber like 'constellation%';`

> 元数据库一般在 `dev_local_xx_all` 中。如果是分库或多个数据中心，可到MC中查看数据中心对应的元数据库实例。

### 问题二：启动升级校验任务异常（MC-000-000）

**现象**：补丁升级失败，报错【MC-000-000】或【启动升级校验任务异常，详情请查看日志】。

**解决方法**：

1. 排查MC的**环境公共配置项**中的 `mc.dm.sort.rule`。
2. **关键检查点**：如果发现配置项中的JSON数据缺少 `"` 引号，请将当前的值修改成带 `"` 的正确JSON格式，再保存回去。
3. 如果之前没有修改过这个配置项，可先备份当前配置值，然后找到对应版本的正确配置值进行替换。
4. 保存后，需要到**集群管理**中发布配置。
5. 重新执行补丁升级。

### 问题三：【MC-003-003】检测到本次升级补丁版本号小于当前应用仓库版本

**现象**：升级补丁时提示检测到本次升级补丁版本号小于当前应用仓库版本。

**原因分析**：一般是升级补丁的顺序有问题。例如没有升级R202408的标准补丁，直接升级了R202408的紧急补丁，然后再升级R202408或R202409的标准补丁。**必须先升级最新版本的标准补丁，才能升级紧急补丁**。直接升级紧急补丁成功了不代表没问题，可能会影响后续升级。

**解决方法**：降版本后重新升级。修改数据库中的应用版本号：

- **MC库**：
  - 应用仓库版本在 `t_mc_appstore_versions` 中，根据集群ID和产品编码查找。
  - 数据中心版本在 `t_mc_datacenterversions` 中，根据数据中心ID和产品编码查找。
- **元数据库**：
  - 对应数据中心的金蝶AI苍穹元数据库 `t_bas_industry_version` 表。
  - 如果是 `cosmic_bos` 的还需要改金蝶AI苍穹元数据库 `t_bas_version` 表。

**把 `fversion` 改为不大于你要升级的那个版本即可**，然后重新按正确顺序升级补丁。

### 问题四：保存配置项提示编码[dubbo.protocol.port]在节点下已存在

**现象**：在管理中心配置环境公共配置项，保存时提示编码 `[dubbo.protocol.port]` 在节点 `[/root/config/common/prop/dubbo.properties]` 下已存在。

**原因分析**：

- 冲突的配置项是历史遗留数据。
- 环境公共配置项点击保存后，会将界面打开的数据行全部保存一遍（即使没有修改）。
- 因此，有冲突配置项的分组在保存时，即使未修改冲突项，也会触发保存并导致冲突校验失败。

**解决方案**：

**方案一**：删除冲突的配置项，再执行配置保存。

**方案二**：绕开冲突检查。保存数据时，程序会保存界面上展示的行数据。如果界面上没有冲突配置项，就不会触发冲突校验。

- **修改配置项操作**：点击搜索框，输入要修改的配置项编码进行过滤，只展示目标配置项，修改后保存。
- **新增配置项操作**：借助同一个目录下的非冲突配置项进行过滤，确保界面上不展示冲突项，再添加新配置项并保存。
